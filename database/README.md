# PII Verification Database

This folder contains the Supabase SQL setup for the collaborative PII verification app.

The database is designed for a team labeling workflow where reviewers should not edit the same JSON file directly. Instead, generated samples and audit findings are imported into Postgres, reviewed through the frontend, protected by locks/version checks, and exported back to JSON when needed.

The intended workflow is Supabase Dashboard first:

1. Open your Supabase project.
2. Go to SQL Editor.
3. Create a new query.
4. Paste and run each SQL file in the order below.

## Run Order

1. `001_schema.sql`
   Creates the database shape: project tables, dataset tables, review tables, audit log table, and indexes.

2. `002_rls_policies.sql`
   Enables Row Level Security, creates membership helper functions, and allows authenticated project members to read only their project data.

3. `003_rpc_functions.sql`
   Creates the safe write API used by the frontend. Review decisions should go through these functions instead of direct table updates.

4. `004_realtime.sql`
   Configures `review_samples` and `review_items` for Supabase Realtime so the UI can react when other reviewers lock or complete rows.

5. `005_seed_project.sql`
   Creates the initial `pii-verification` project and adds your first Supabase Auth user as owner.

6. `006_multi_entity_datasets.sql`
   Optional migration only for databases that already ran the older one-class-per-dataset schema. Fresh setups do not need this file because `001_schema.sql` already includes the new shared-source model.

## File Details

### `001_schema.sql`

Purpose: create the core relational model.

This file creates:

- `public.labeling_projects`
- `public.project_members`
- `public.datasets`
- `public.review_samples`
- `public.review_items`
- `public.audit_events`
- indexes for common query paths
- `private` schema for helper functions created later

Run this first because every other SQL file depends on these tables.

### `002_rls_policies.sql`

Purpose: protect rows by project membership.

This file creates:

- `private.is_project_member(project_id)`
- `private.can_edit_project(project_id)`
- RLS policies for all public tables
- read-only grants for authenticated users

Frontend users can select rows only when they belong to the project through `project_members`. Direct table writes are intentionally not granted to the frontend. Writes are handled by RPC functions in `003_rpc_functions.sql`.

### `003_rpc_functions.sql`

Purpose: provide safe write operations for the frontend.

This file creates:

- `public.acquire_sample_lock`
- `public.release_sample_lock`
- `public.submit_review_decision`

These RPC functions enforce:

- authenticated user checks,
- project role checks,
- sample lock ownership,
- optimistic version checks,
- atomic sample/item updates,
- audit logging.

This is the main protection against overwrite conflicts when multiple reviewers work at the same time.

### `004_realtime.sql`

Purpose: let the frontend subscribe to collaborative changes.

This file:

- sets `replica identity full` on `review_samples` and `review_items`,
- adds both tables to the `supabase_realtime` publication when available.

The frontend can then subscribe to row updates and show states like:

- another reviewer locked this sample,
- another reviewer completed this item,
- your local row version is stale and should be refreshed.

### `005_seed_project.sql`

Purpose: create the first project and owner membership.

Before running it:

1. Open Supabase Dashboard.
2. Go to Authentication -> Users.
3. Create or invite your first user.
4. Copy that user's UUID.
5. Replace `<your-auth-user-id>` in `005_seed_project.sql`.

The script uses `on conflict` so it can be re-run safely during setup.

### `006_multi_entity_datasets.sql`

Purpose: migrate an already-created database from one dataset per class to one dataset per output/source file.

Run it once if you already applied an older `001_schema.sql` before this change. It adds `datasets.source_key`, adds `review_items.entity_type`, replaces the old dataset uniqueness rule, and rebuilds the review item uniqueness index so different classes can share the same sample rows.

## Table Schemas

### `labeling_projects`

Top-level workspace for labeling work. For this app, you may start with a single project named `PII Verification`.

| Column | Type | Purpose |
| --- | --- | --- |
| `id` | `uuid` | Primary key generated by Postgres. |
| `slug` | `text` | Stable URL/code-friendly identifier, for example `pii-verification`. Must be unique. |
| `name` | `text` | Human-readable project name. |
| `description` | `text` | Optional project description. |
| `created_by` | `uuid` | Optional reference to `auth.users(id)` for the creator. |
| `created_at` | `timestamptz` | Creation timestamp. |
| `archived_at` | `timestamptz` | Optional timestamp for hiding old projects without deleting them. |

Relationships:

- Referenced by `project_members.project_id`.
- Referenced by `datasets.project_id`.
- Referenced by `audit_events.project_id`.

### `project_members`

Access-control table connecting Supabase Auth users to projects.

| Column | Type | Purpose |
| --- | --- | --- |
| `project_id` | `uuid` | Project membership belongs to. |
| `user_id` | `uuid` | Supabase Auth user. |
| `role` | `text` | One of `owner`, `admin`, `reviewer`, `viewer`. |
| `created_at` | `timestamptz` | Membership creation timestamp. |

Primary key:

- `(project_id, user_id)`

Role meaning:

- `owner`: full project owner. Use for first/admin user.
- `admin`: can manage project data in future admin UI.
- `reviewer`: can review and submit decisions.
- `viewer`: can read project data but should not edit.

Current RPC edit access is granted to `owner`, `admin`, and `reviewer` through `private.can_edit_project`.

### `datasets`

One imported output/source batch. A dataset now represents the shared sample file, not a single PII class. Multiple classes such as `ACCOUNT_NUMBER`, `EMAIL`, and `PERSON_NAME` attach to the same dataset through `review_items.entity_type`.

Example dataset:

- source key: `en/result1_train_verified`
- entity type: `MULTI_ENTITY`
- language: `en`
- folder: `1`
- source output file stored inside `metadata.source_files`

| Column | Type | Purpose |
| --- | --- | --- |
| `id` | `uuid` | Primary key. |
| `project_id` | `uuid` | Owning project. |
| `source_key` | `text` | Stable source identifier used as the sample key prefix, for example `en/result1_train_verified`. |
| `entity_type` | `text` | Dataset scope marker. New imports use `MULTI_ENTITY`; the actual reviewed PII class lives on `review_items.entity_type`. |
| `language` | `text` | Dataset language such as `en`, `vi`, or `de`. |
| `folder` | `text` | Original local folder/batch identifier. |
| `metadata` | `jsonb` | Extra import metadata, source filenames, counts, source paths, etc. |
| `created_by` | `uuid` | Optional Auth user who imported the dataset. |
| `created_at` | `timestamptz` | Dataset creation timestamp. |

Unique index:

- `datasets_source_unique` on `(project_id, language, coalesce(folder, ''), source_key)`

Recommended `metadata` shape:

```json
{
  "source_files": {
    "output": "result1_train_verified.json"
  },
  "import": {
    "source": "verify_entity import",
    "source_key": "en/result1_train_verified",
    "counts": {
      "output_samples": 1628
    }
  },
  "latest_entity_import": {
    "entity_type": "ACCOUNT_NUMBER",
    "export_summary": {
      "total_occurrences": 113
    }
  }
}
```

Why filenames live in `metadata`: they are useful for tracing/debugging but are not core relational fields used by review workflows. Keeping them in JSON keeps the schema smaller while preserving source provenance.

### `review_samples`

One row per generated output sample. This is the lockable unit.

A sample is the full text plus its full `privacy_mask`. We lock at sample level because a decision can change text length and shift offsets for multiple mask entries.

| Column | Type | Purpose |
| --- | --- | --- |
| `id` | `uuid` | Primary key. |
| `dataset_id` | `uuid` | Parent dataset. |
| `sample_index` | `integer` | Original array index in the output JSON. |
| `sample_key` | `text` | Stable key such as `en/result1_train_verified#59`. |
| `language` | `text` | Sample language. |
| `original_source_text` | `text` | Source text exactly as imported. Never edited. |
| `current_source_text` | `text` | Latest reviewed source text. Export uses this column. |
| `original_privacy_mask` | `jsonb` | Original `privacy_mask` exactly as imported. Never edited. |
| `current_privacy_mask` | `jsonb` | Latest reviewed `privacy_mask`. Export uses this column. |
| `raw_output` | `jsonb` | Full original output sample object for debugging. |
| `version` | `integer` | Optimistic concurrency counter. Incremented on save. |
| `locked_by` | `uuid` | Auth user currently holding the lock. |
| `locked_until` | `timestamptz` | Lock expiration timestamp. |
| `updated_by` | `uuid` | Last Auth user who changed reviewed sample state. |
| `updated_at` | `timestamptz` | Last update timestamp. |
| `created_at` | `timestamptz` | Creation timestamp. |

Unique constraints:

- `(dataset_id, sample_index)`
- `(dataset_id, sample_key)`

Common UI usage:

- list samples in original order,
- show lock owner/expiration,
- submit updated source text and privacy mask,
- export final JSON.

### `review_items`

One row per audit finding/span that needs a human decision.

A single sample can have many review items. For example, one text may contain several values flagged as `WRONG_LABEL`.

| Column | Type | Purpose |
| --- | --- | --- |
| `id` | `uuid` | Primary key. |
| `dataset_id` | `uuid` | Parent dataset. |
| `sample_row_id` | `uuid` | Parent `review_samples` row. |
| `sample_key` | `text` | Duplicated stable sample key for easy lookup/debugging. |
| `entity_type` | `text` | PII class for this finding, for example `ACCOUNT_NUMBER` or `EMAIL`. |
| `audit_record_id` | `integer` | Original `id` from audit result. |
| `value` | `text` | Entity value being reviewed. |
| `start_offset` | `integer` | Start offset from export/context JSON when available. |
| `end_offset` | `integer` | End offset from export/context JSON when available. |
| `verdict` | `text` | Audit verdict: `CORRECT`, `WRONG_LABEL`, or `UNREALISTIC_VALUE`. |
| `reason` | `text` | Audit explanation. |
| `suggested_label` | `text` | Suggested replacement label for wrong-label cases. |
| `replacement_value` | `text` | Suggested replacement value for unrealistic-value cases. |
| `status` | `text` | Review state: `pending`, `completed`, or `skipped`. |
| `decision` | `text` | Human decision: `accept`, `deny`, `deny_keep`, or `deny_remove`. |
| `reviewer_note` | `text` | Optional human note. |
| `decided_by` | `uuid` | Auth user who submitted the decision. |
| `decided_at` | `timestamptz` | Decision timestamp. |
| `raw_audit` | `jsonb` | Full original audit object. |
| `raw_export_span` | `jsonb` | Matching export/context span object. |
| `version` | `integer` | Optimistic concurrency counter for the item. |
| `updated_at` | `timestamptz` | Last update timestamp. |
| `created_at` | `timestamptz` | Creation timestamp. |

Unique index:

- `review_items_unique_span` on `(dataset_id, entity_type, sample_key, audit_record_id, value, coalesce(start_offset, -1), coalesce(end_offset, -1))`

Common UI usage:

- filter pending/completed items,
- search by value/reason/suggested label/source context,
- show audit verdict and context,
- save the human decision.

### `audit_events`

Append-only history table for collaborative safety and debugging.

| Column | Type | Purpose |
| --- | --- | --- |
| `id` | `bigint` | Sequential primary key. |
| `project_id` | `uuid` | Related project. |
| `dataset_id` | `uuid` | Related dataset. |
| `sample_row_id` | `uuid` | Related sample when applicable. |
| `review_item_id` | `uuid` | Related review item when applicable. |
| `actor_id` | `uuid` | Auth user who performed the action. |
| `action` | `text` | Action name such as `lock_acquired`, `lock_released`, or `decision_submitted`. |
| `before_state` | `jsonb` | JSON snapshot before action. |
| `after_state` | `jsonb` | JSON snapshot after action. |
| `created_at` | `timestamptz` | Event timestamp. |

Use this table when you need to answer questions like:

- Who changed this sample?
- When was this decision submitted?
- What did the privacy mask look like before the change?

## Helper And RPC Functions

### `private.is_project_member(project_id)`

Returns `true` when the current authenticated user has a row in `project_members` for the given project.

Used by:

- RLS policies on `labeling_projects`,
- RLS policies on `datasets`,
- RLS policies on `review_samples`,
- RLS policies on `review_items`,
- RLS policies on `audit_events`.

### `private.can_edit_project(project_id)`

Returns `true` when the current authenticated user is a project `owner`, `admin`, or `reviewer`.

Used by:

- `acquire_sample_lock`,
- `release_sample_lock`,
- `submit_review_decision`.

This is why viewers can read data but cannot submit review changes.

### `public.acquire_sample_lock(sample_id, expected_version, ttl_seconds)`

Locks one `review_samples` row for the current user.

Default TTL is 180 seconds. Requested TTL values are clamped to 30-300 seconds because reviewers usually finish one item quickly, but dead locks should also expire fast.

It succeeds only when:

- the user is authenticated,
- the user can edit the project,
- the sample row still has the expected version,
- the sample is unlocked, already locked by the same user, or the old lock has expired.

It writes a `lock_acquired` audit event when the lock owner changes.

### `public.release_sample_lock(sample_id)`

Releases a sample lock if the current user owns it.

Use this when:

- the reviewer leaves the active review item/session,
- the reviewer leaves the page,
- the frontend wants to clean up an abandoned edit session.

It writes a `lock_released` audit event when a lock is actually released.

### `public.submit_review_decision(...)`

Submits a human review decision and updates the parent sample in one database transaction.

It checks:

- authenticated user,
- edit permission,
- item version,
- sample version,
- current user owns the sample lock,
- lock has not expired,
- `decision` is one of the allowed values,
- `new_privacy_mask` is a JSON array when provided.

It updates:

- `review_items.status`,
- `review_items.decision`,
- `review_items.reviewer_note`,
- `review_items.decided_by`,
- `review_items.decided_at`,
- `review_items.version`,
- `review_samples.current_source_text`,
- `review_samples.current_privacy_mask`,
- `review_samples.version`,
- `review_samples.updated_by`,
- `review_samples.updated_at`.

It also releases the sample lock and writes a `decision_submitted` audit event.

## Realtime Behavior

`004_realtime.sql` prepares `review_samples` and `review_items` for Supabase Realtime subscriptions.

Recommended frontend subscriptions:

- `review_samples` updates for lock state, sample version, source text, and privacy mask changes.
- `review_items` updates for completed decisions and status changes.

Use Realtime as a UI synchronization layer, not as the source of truth. The source of truth is still Postgres plus the RPC version checks.

## Security Model

Frontend users can read rows only when they are members of the project.

Frontend users should not directly insert, update, or delete table rows. Writes should go through the RPC functions in `003_rpc_functions.sql` so the database can enforce:

- authenticated user checks,
- project role checks,
- sample locking,
- optimistic version checks,
- atomic updates,
- audit logging.

Admin import/export scripts should use a service role key locally or in secure CI only. Never expose the service role key in frontend code or public deployment variables.

## Re-running SQL

Most files are safe to re-run during setup:

- Functions use `create or replace`.
- Policies are dropped before being recreated.
- Realtime publication setup checks whether tables are already added.
- The seed file uses `on conflict` for the project and owner membership.

`001_schema.sql` uses `create table if not exists`, but it is still intended as an initial schema setup file. If you already ran the older one-class-per-dataset schema, run `006_multi_entity_datasets.sql` once to add the shared-source fields and indexes.

### If You Already Ran An Older Dataset Schema

An older draft of `datasets` had three filename columns:

- `output_file_name`
- `export_file_name`
- `audit_file_name`

The current schema stores those values in `metadata.source_files` instead. If you already ran the old schema and want to migrate without losing existing filename values, run this once before dropping the old columns:

```sql
update public.datasets
set metadata = jsonb_set(
  coalesce(metadata, '{}'::jsonb),
  '{source_files}',
  jsonb_build_object(
    'output', output_file_name,
    'export', export_file_name,
    'audit', audit_file_name
  ),
  true
)
where output_file_name is not null
   or export_file_name is not null
   or audit_file_name is not null;

alter table public.datasets drop column if exists output_file_name;
alter table public.datasets drop column if exists export_file_name;
alter table public.datasets drop column if exists audit_file_name;
```
